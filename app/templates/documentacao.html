Documentacao do projeto

Guia criado por Max para ajudar quem for continuar o projeto depois. A ideia aqui e mostrar onde mexer, o que cada arquivo faz e o que tomar cuidado para nao quebrar outra parte sem querer.

Se quiser abrir essa pagina depois, use /documentacao no navegador.

1. Nomes dos arquivos e o que pode ignorar

Os nomes principais do projeto estao normais e simples. Nao achei nada com cara de nome automatico estranho tipo arquivo de robo.

Arquivos reais do projeto

  • run.py abre o servidor.
  • app/templates/*.html sao as paginas.
  • app/static/css/*.css sao os visuais.
  • app/static/js/*.js sao as interacoes.
  • app/routes, controllers, services e models sao o backend.

Arquivos gerados automaticamente

  • __pycache__ e criado pelo Python.
  • server.out.log e server.err.log aparecem quando o servidor e iniciado pelo terminal.
  • app/database/doa_ja.db e o banco local gerado a partir do SQL.

Arquivos de referencia

  • arquivos novos/ guarda o material original usado como base visual.
  • estrutura.html pode servir de apoio, mas nao e a pagina usada pelo Flask.

3. Arquivos raiz

run.py

Esse arquivo e o ponto de partida. Quando alguem roda python run.py, e ele que chama a aplicacao Flask.

  • app = create_app(): cria a aplicacao principal.
  • app.run(...): sobe o servidor local.

Se precisar trocar host, porta ou debug, o lugar mais facil e aqui.

requirements.txt

Guarda as bibliotecas do projeto. Hoje a principal e o Flask.

Se instalar algo novo, adiciona aqui para o outro integrante conseguir rodar tambem.

readme.md

Explica como rodar o projeto e quais usuarios de teste existem. Sempre atualize esse arquivo quando mudar a forma de iniciar ou os logins.

estrutura.html e pasta arquivos novos

Esses arquivos servem de referencia visual. O projeto rodando no navegador nao usa direto esses arquivos. Se mudar alguma coisa neles, a tela real do Flask nao muda automaticamente.

4. Arquivo principal do app

app/main.py

create_app(): cria o Flask, coloca a chave da sessao, inicia o banco e registra as blueprints.

Se uma rota nova nao estiver aparecendo, normalmente o erro esta aqui ou no arquivo de rotas que nao foi importado.

5. Rotas

Os arquivos de rota so servem para ligar uma URL a uma funcao do controller. Eles costumam ser curtos.

app/routes/home_routes.py
  • index(): abre a pagina inicial.
  • dashboard(): abre o dashboard.
  • chat(): abre o chat.
  • profile(): abre o perfil do usuario logado.
  • public_profile(usuario_id): abre o perfil publico de um usuario.
  • documentation(): abre esta pagina de documentacao.
  • admin_dashboard_redirect(): manda para a area admin.
app/routes/auth_routes.py
  • login(): rota de entrar.
  • register(): rota de cadastro.
  • logout(): rota para sair da conta.
app/routes/item_routes.py
  • items(): lista geral dos anuncios.
  • item_detail(item_id): abre um produto.
  • create_item(): tela e envio do anuncio.
  • my_items(): mostra os anuncios do usuario.
app/routes/request_routes.py
  • request_item(item_id): envia pedido de doacao.
  • my_requests(): lista pedidos feitos.
app/routes/admin_routes.py
  • dashboard(): painel principal do admin.
  • users(): lista usuarios.
  • items(): lista anuncios.
  • requests(): lista pedidos.

6. Controllers

Controller recebe a rota e decide qual pagina abrir, qual service chamar e para onde redirecionar.

app/controllers/home_controller.py
  • index(): pega os itens e manda para a home.
  • dashboard(): exige login, monta resumo e listas do usuario.
  • chat(): abre a pagina do chat.
  • profile(): pega dados do usuario logado.
  • public_profile(usuario_id): pega dados de outro usuario.
  • documentation(): abre esta pagina HTML.

Se uma tela estiver abrindo sem dados ou com dados errados, o primeiro lugar para olhar costuma ser aqui.

app/controllers/auth_controller.py
  • login(): verifica se ja esta logado, recebe email e senha do formulario, chama o service e salva a sessao.
  • register(): recebe os dados do cadastro e chama o service para salvar.
  • logout(): limpa a sessao.

Cuidado com as chaves da sessao: user_id, user_name e user_type. Elas controlam o menu e as permissoes.

app/controllers/item_controller.py
  • items(): manda a lista geral para a pagina.
  • item_detail(item_id): busca um item pelo id.
  • create_item(): trata o formulario de anuncio.
  • my_items(): traz os anuncios do usuario.

Se o formulario de anuncio parar de salvar, olha primeiro esse arquivo e depois o item_service.py.

app/controllers/request_controller.py
  • request_item(item_id): recebe a mensagem e pede a doacao.
  • my_requests(): lista os pedidos do usuario.
app/controllers/admin_controller.py
  • dashboard(): mostra resumo do admin.
  • users(): abre tabela de usuarios.
  • items(): abre tabela de anuncios.
  • requests(): abre tabela de pedidos.

Esse arquivo sempre checa se a pessoa logada e admin antes de abrir as telas.

7. Services

Service e a parte das regras. Ele valida, arruma os dados e chama o model.

app/services/auth_service.py
  • login_user(email, senha): valida campos e procura o usuario no banco.
  • register_user(...): valida nome, email, senha, tipo e localizacao.
  • get_user_by_id(user_id): pega um usuario pelo id.

Se quiser deixar senha mais forte ou mudar as validacoes, mexa aqui.

app/services/item_service.py
  • list_items(): lista todos os itens.
  • get_item_by_id(item_id): pega um item.
  • list_items_by_user(user_id): pega itens de um usuario.
  • create_item(formulario, user_id, user_name): valida e salva anuncio novo.
  • get_dashboard_data(): calcula total, doados e restantes.

Cuidado com as variaveis nome, localizacao, foto, descricao, doador_nome, categoria e avaliacao. Elas vem do formulario.

app/services/request_service.py
  • create_request(item_id, user_id, mensagem): impede pedir item proprio, impede item indisponivel e impede pedido repetido.
  • list_requests_by_user(user_id): lista pedidos do usuario.
  • count_requests_by_user(user_id): conta pedidos.
app/services/admin_service.py
  • get_admin_data(): pega os numeros do painel admin.
  • list_users(): lista usuarios.
  • list_items(): lista anuncios.
  • list_requests(): lista pedidos.

8. Models

Model e a parte que conversa direto com o banco SQLite.

app/models/user_model.py
  • get_user_by_email_and_password(email, senha): busca usuario para login.
  • get_user_by_email(email): busca por email.
  • get_user_by_id(user_id): busca por id.
  • list_users(): lista usuarios.
  • create_user(...): salva usuario novo.
app/models/item_model.py
  • list_items(): lista anuncios com nome do dono.
  • get_item_by_id(item_id): busca anuncio pelo id.
  • list_items_by_user(user_id): lista anuncios de um usuario.
  • create_item(...): salva no banco.
  • count_all_items(): conta todos.
  • count_donated_items(): conta status doado.
  • count_available_items(): conta disponivel.

Se mudar nome de coluna no SQL, esse arquivo quase sempre precisa mudar tambem.

app/models/donation_request_model.py
  • create_request(...): salva pedido.
  • find_request(...): ve se ja existe pedido igual.
  • list_requests_by_user(user_id): lista pedidos do usuario.
  • list_all_requests(): lista tudo para o admin.
  • count_requests_by_user(user_id): conta pedidos.
app/models/admin_model.py

get_admin_data(): conta usuarios, anuncios e pedidos para mostrar no painel.

9. Banco e arquivos auxiliares

app/database/connection.py
  • get_connection(): abre conexao com o SQLite.
  • init_database(): le o schema.sql e cria o banco se precisar.

Se o banco quebrar, apaga o arquivo app/database/doa_ja.db e sobe o projeto de novo.

app/database/schema.sql

Cria as tabelas usuarios, anuncios e solicitacoes. Tambem deixa usuarios e anuncios iniciais para teste.

Se quiser mudar os cards da home quando o banco for recriado, mexa nos INSERT INTO anuncios daqui.

app/utils/validators.py
  • text_is_filled(texto): ve se o campo esta vazio.
  • email_is_valid(email): faz validacao simples de email.
app/utils/helpers.py
  • normalize_text(valor): limpa espacos e evita None.
  • get_default_image(link): escolhe imagem padrao se o campo vier vazio.
app/utils/security.py
  • user_is_logged(): ve se tem usuario na sessao.
  • user_is_admin(): ve se o tipo e admin.

10. Templates HTML

Esses arquivos montam as paginas. Se quiser mudar titulo, textos, botoes, ordem dos blocos ou nomes exibidos na tela, normalmente o lugar e aqui.

Arquivo Uso O que mexer aqui Cuidado
base.html Layout base com navbar, footer e scripts. Muda menu, logo, rodape e estrutura comum. Se tirar um bloco do base, afeta todas as paginas.
index.html Pagina principal. Barra de busca, filtros e cards da home. Os filtros usam o atributo data-filter. Se mudar o nome da categoria, muda no banco e no JS tambem.
login.html Tela de login. Campos, textos, botoes e usuarios de teste exibidos. Os nomes dos inputs precisam continuar como email e senha.
register.html Tela de cadastro. Campos do formulario de registro. Os nomes dos inputs precisam bater com o controller.
dashboard.html Painel do usuario. Cards de resumo e listas do usuario. Usa variaveis resumo, meus_itens e meus_pedidos.
create_item.html Formulario de anuncio. Campos, labels e preview da foto. Os names do form precisam continuar iguais aos que o service espera.
item_list.html Lista geral de produtos. Visual dos cards fora da home. Usa a variavel itens.
item_detail.html Detalhe do produto. Descricao, bloco do doador e formulario de pedido. Usa a variavel item. Se mudar nome de campo do banco, ajusta aqui.
my_items.html Lista dos anuncios do usuario. Visual da grade de anuncios proprios. Usa a variavel itens.
my_requests.html Lista de pedidos. Cards dos pedidos feitos. Usa a variavel pedidos.
chat.html Chat visual. Estrutura da conversa e lista lateral. O envio visual depende dos ids chat-form, chat-input e chat-messages.
profile.html Perfil do usuario e perfil publico. Nome, bio, tags e lista de anuncios. Usa usuario, itens e publico.
admin/*.html Telas da area admin. Tabelas e numeros do painel. Qualquer mudanca aqui costuma depender tambem do admin controller e admin service.

11. CSS

app/static/css/style.css

E o CSS principal do site. Ele controla navbar, home, cards, dashboard, perfil, admin e chat.

  • .navbar: menu do topo.
  • .search-section: bloco da busca da home.
  • .filter-btn: botoes de filtro.
  • .home-card: card da home no estilo do modelo.
  • .card: card base usado em outras paginas.
  • .badge-*: cores dos status.

Se quiser mudar o visual da pagina principal, comece por esse arquivo e pelo index.html.

app/static/css/auth.css

Controla login e registro.

  • .auth-body: fundo da tela.
  • .auth-card: cartao branco central.
  • .input-group: grupo de campo com icone.
  • .btn-primary-auth: botao laranja.

Se login e registro mudarem muito, provavelmente esse arquivo e o login.html vao ser editados juntos.

12. JavaScript

app/static/js/app.js

Esse JS cuida do menu mobile, da busca, dos filtros, do preview da foto e do chat visual.

  • mobileMenu: botao do menu em tela pequena.
  • navLinks: lista de links da navbar.
  • searchInput: input da busca da home.
  • filterButtons: botoes das categorias.
  • cards: cards da home filtrados pelo JS.
  • currentFilter: categoria atual selecionada.
  • updateItems(): funcao que esconde e mostra cards.
  • fotoInput e fotoPreview: atualizam a imagem do anuncio enquanto digita.
  • chatForm, chatInput e chatMessages: fazem o chat visual funcionar.

Cuidado para nao mudar ids do HTML sem ajustar esse arquivo.

app/static/js/auth.js

So controla o botao de mostrar ou esconder senha.

  • buttons: botoes com classe toggle-password.
  • data-target: id do input que sera alterado.

Se mudar o id do campo de senha no HTML, muda o data-target junto.

13. Arquivos simples de organizacao

Os arquivos __init__.py em varias pastas servem so para o Python reconhecer a pasta como pacote. Eles nao tem regra de negocio.

Os arquivos .gitkeep servem so para manter pasta vazia no Git. Tambem nao tem logica.

14. Dicas para quem vai editar

  • Para mudar visual da home: mexa primeiro em app/templates/index.html e app/static/css/style.css.
  • Para mudar o menu inteiro: mexa em app/templates/base.html.
  • Para mudar login e cadastro: mexa em login.html, register.html e auth.css.
  • Para mudar usuarios de teste ou itens iniciais: mexa em app/database/schema.sql e depois recrie o banco.
  • Se adicionar campo novo no banco, normalmente precisa mudar SQL, model, service, controller e template.
  • Se uma pagina abre mas sem dado, olha a ordem: rota, controller, service, model, template.
